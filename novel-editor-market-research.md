# 小說編輯器深度市場研究報告
> 目標市場：兩岸三地華文作者（涵蓋網文連載 + 傳統文學）
> 主要競爭對手：Scrivener、Notion/Obsidian、各大平台內建編輯器
> 研究日期：2026-06-19

---

## 一、小說編輯的流程分析

### 1.1 從草稿到出版：五大階段

| 階段 | 別名 | 核心任務 | 工具需求 |
|------|------|---------|---------|
| **發想期** | Brainstorming | 角色概念、世界觀、靈感收集 | 快速筆記、視覺化板 |
| **構架期** | Outlining | 大綱、章節結構、時間線 | 層級式大綱、卡片板 |
| **草稿期** | First Draft | 高速碼字，不回頭 | 無干擾模式、字數追蹤 |
| **自修期** | Self-editing | 多輪自我審閱與修改 | 版本快照、批注、搜尋取代 |
| **校稿期** | Professional Editing | 結構→行文→語法→最終校對 | 協作批注、格式輸出 |

### 1.2 三大作者派別與工具需求差異

#### 🗓 Plotter（計畫派）
- **特色**：寫作前把故事框架規劃清楚，章節任務明確後才動筆
- **比例**：約佔作者群 30–40%
- **工具重點**：大綱工具、卡片板（Corkboard）、人物弧線追蹤、情節模板（英雄旅程 / 雪花法等）
- **代表工具**：Scrivener（軍事級大綱）、Novelcrafter（Codex + 場景管理）

#### ✍️ Pantser（直覺派）
- **特色**：「坐下來就開始寫」，靠直覺發展故事，幾乎不預先規劃
- **比例**：約佔 30–40%，網文作者比例更高
- **工具重點**：快速開啟即寫、無干擾模式、自動儲存、雲端同步（手機 / 電腦切換）
- **代表工具**：純純寫作、Word、Google Docs

#### 🔀 Planster（折衷派）
- **特色**：粗略大綱 + 彈性發展，邊寫邊調整結構
- **比例**：最大群體，約 30–40%
- **工具重點**：大綱與寫作視窗並存、場景標籤狀態、不干擾流的設定調整

---

### 1.3 傳統文學 vs 網文的工作流程差異

| 維度 | 傳統文學作者 | 網文 / 連載作者 |
|------|------------|--------------|
| **更新節奏** | 完成後投稿/出版 | 每日 2,000–6,000 字更新 |
| **寫作壓力** | 來自創作品質 | 來自日更、讀者催更 |
| **存稿習慣** | 幾乎沒有 | 存稿 10–30 章是基本生存法則 |
| **大綱細緻度** | 高（故事邏輯需完整） | 低（靈活應對讀者反應） |
| **字數目標** | 單書 8–12 萬字 | 單書 100–500 萬字起跳 |
| **設定複雜度** | 中（角色為主） | 高（系統、功法、世界觀複雜） |
| **校稿重視度** | 高（出版前多輪） | 低（發布後才修改） |
| **平台整合需求** | 低（投稿 Word 檔） | 高（直接在平台提交章節） |

---

### 1.4 專業校稿的五個層次

根據業界標準，一部稿件從草稿到出版需經過：

1. **發展性編輯（Developmental Editing）**
   - 最耗時的階段；審查大方向：情節結構、人物弧線、節奏、類型適配
   - 產出：編輯信（Editorial Letter），作者需要大幅修改

2. **稿件評估（Manuscript Review）**
   - 發展性編輯的輕量版；告知主要問題但不提供逐章解方
   - 適合預算有限者或已有多輪 beta reading 的作者

3. **行文編輯（Line Editing）**
   - 聚焦句子層面：措辭選擇、句型節奏、段落流動、聲音一致性
   - 不處理語法錯別字，而是處理「說清楚了嗎？」

4. **文字編輯（Copy Editing）**
   - 清理語法錯誤、標點符號、格式一致性、引文正確性
   - 嚴格遵循格式規範（如：台灣出版業的書名號使用習慣）

5. **最終校對（Proofreading）**
   - 排版後的最後一道防線；捕捉 copy editing 漏掉的細節
   - 不作改寫，只找 typo 與排版錯誤

> **網文作者現實**：大多數連載作者跳過 2–4 步，直接從草稿到發布。

## 二、競爭對手功能分析

### 2.1 Scrivener（行業標竿，一次性付費）

**定位**：桌面端重度寫作工具，業界超過 20 年歷史

| 功能模組 | 內容 |
|---------|------|
| **Binder（書架）** | 階層式文件管理，可無限嵌套資料夾 |
| **Corkboard（軟木板）** | 索引卡視圖，拖拉排列場景/章節 |
| **Outliner 視圖** | 表格式大綱，顯示字數、狀態、摘要 |
| **Synopsis（摘要欄）** | 每份文件附摘要卡，對照全文結構 |
| **Composition Mode** | 全螢幕無干擾寫作模式 |
| **字數目標** | 可設書籍總目標 + 每日目標，進度條顯示 |
| **Research 資料夾** | 可存圖片、PDF、網頁快照等研究資料 |
| **Labels & Status** | 可為場景標注進度（初稿/修改中/完成） |
| **Snapshots（快照）** | 每份文件可儲存歷史版本 |
| **Compile（輸出）** | 可輸出至 DOCX、EPUB、PDF、Markdown 等多格式 |
| **Comments & Annotations** | 行內批注與標注系統 |
| **Split Editor** | 同時顯示兩份文件（如：大綱 + 正文） |

**Scrivener 的核心優勢：**
- 「一個專案容納一切」的哲學：文稿 + 研究資料 + 大綱 + 角色資料全在一起
- Corkboard 讓場景重組變得直觀
- Compile 系統處理複雜的格式排版需求

**Scrivener 的痛點（用戶高頻抱怨）：**
- 學習曲線極高，新手 7–14 天才能上手
- UI 設計老舊（雖 2024 年後有改版，但整體仍較凌亂）
- 行動端（iOS/Android）體驗差，同步機制複雜
- 與平台投稿工作流程脫節（網文作者不需要 Compile）
- 一次性付費 $59 USD，試用後猶豫率高

---

### 2.2 Notion / Obsidian（靈活通用型）

| 維度 | Notion | Obsidian |
|------|--------|---------|
| **定位** | 雲端協作、萬用筆記 | 本地 Markdown、知識管理 |
| **離線支援** | ❌（需網路） | ✅（完全離線） |
| **章節管理** | 自建（無原生小說結構） | 自建（資料夾即結構） |
| **字數追蹤** | ❌ 原生不支援 | ❌ 需安裝插件 |
| **無干擾寫作** | 有（Focus Mode）但不強 | ✅ 精簡模式好用 |
| **世界觀管理** | ✅ 雙向連結、資料庫 | ✅ 雙向連結、圖譜 |
| **費用** | 免費 / $10/月（Pro） | 免費（個人）/ $10/月（Sync） |
| **最大弱點** | 純寫作體驗差；作者需大量自建模板 | 無內建小說結構；學習曲線高 |

**為何作者轉向 Notion/Obsidian：**
- 對 Scrivener 學習曲線感到挫折
- 需要良好的世界觀/設定 Wiki 系統
- 跨設備同步比 Scrivener 簡單

**核心缺口：** 兩者都缺乏「專門為小說設計的字數目標 + 連載進度追蹤」

### 2.3 中國大陸網文平台內建編輯器（起點 / 番茄 / 晉江）

| 功能 | 支援狀況 |
|------|---------|
| **基本文字編輯** | ✅ |
| **章節管理** | ✅（簡單列表） |
| **字數顯示** | ✅ |
| **防抄襲保護** | ✅ |
| **平台直投** | ✅（最強點） |
| **大綱工具** | ❌ |
| **人物管理** | ❌ |
| **版本快照** | ❌ |
| **無干擾模式** | ❌ |
| **跨平台離線** | ❌ |

**結論：** 平台編輯器是工具，不是工作空間。網文作者最大的抱怨是「只能在一個平台寫，換平台就很麻煩」。

## 三、用戶高度評價的功能總整理

### 3.1 作者最常提到的「改變了我的工作流」功能

以下基於多份研究（Reddit、Substack 評測、中文論壇、App Store 評論）整理：

| 功能 | 重要性來源 | 推崇原因 |
|------|-----------|---------|
| **無干擾寫作模式** | 全球作者共識 | 「開了瀏覽器就完了」；心流狀態的最大殺手 |
| **字數目標 + 進度條** | Scrivener 標誌功能 | 量化讓動力具體化；每日目標感提升完成率 |
| **章節自由排列（拖拉）** | Scrivener Corkboard | 打破線性寫作限制，結構優化變直觀 |
| **自動儲存 + 防丟稿** | 純純寫作代表性功能 | 對抗「一次意外丟稿等於放棄寫作」的心理創傷 |
| **雲端同步（電腦↔手機）** | 網文作者核心需求 | 通勤/睡前用手機，回家用電腦；切換無縫才能高產 |
| **快照/版本歷史** | Scrivener 核心功能 | 改壞了能回頭；讓作者敢大幅修改 |
| **人物管理（資料卡）** | Scrivener / Novelcrafter | 記不住自己寫的人物是真實痛點；百萬字長篇必需 |
| **分割視窗（大綱+正文）** | 進階用戶需求 | 寫作時能看到人物資料/大綱，不用切換視窗 |
| **場景標籤（狀態標注）** | Plotter 的最愛 | 標記「初稿/修改中/完成」讓修改流程有序 |
| **字數統計細節** | 網文作者剛需 | 今日新增、本章、全書、更新週期字數分開計算 |
| **沉浸式 UI / 美觀設計** | 出現在22款工具評測中 | 「待在裡面寫作的感覺」影響創作動力 |
| **離線寫作** | Obsidian 標誌優勢 | 「一打開瀏覽器就被分心帶走」 |
| **語音輸入** | 中文市場特有重點 | 通勤/散步時的靈感輸入；手速解放 |
| **出口格式多元** | 自出版作者需求 | 不被工具綁架；DOCX / EPUB / TXT |

### 3.2 被高度推崇但常被競品忽略的功能

- **「寫作連續天數」Streak 追蹤**：GameBoy 式的習慣養成動力
- **Beta 閱讀連結分享**：讓讀者/校稿者可以讀稿但不下載
- **章節摘要卡（Synopsis）**：每章附一句話摘要，讓結構一眼可看
- **全文搜尋 + 取代**：一本百萬字小說裡找一個錯誤人名，沒搜尋是噩夢
- **時間軸視圖**：複雜多線敘事的必需品
- **讀者統計整合**：網文作者想知道哪章掉訂最多

## 四、功能優先級：Must-Have vs Good-to-Have

> 以下分析基於「兩岸三地華文作者（網文 + 傳統文學）」為核心用戶群。

---

### ✅ MUST HAVE（核心體驗，沒有就會流失）

#### 🖊 寫作核心體驗

| 功能 | 說明 | 判斷依據 |
|------|------|---------|
| **無干擾寫作模式** | 隱藏所有 UI 元素，全螢幕純文字 | 100% 用戶評測均提到；心流條件 |
| **自動儲存 + 衝突偵測** | 永遠不丟稿，衝突時提示 | 純純寫作因此成功；丟稿是最大流失原因 |
| **雲端同步（手機↔電腦）** | 即時或近即時的跨裝置同步 | 網文作者的基礎設施需求 |
| **良好的中文輸入體驗** | 全形標點、自動縮排、不亂跳游標 | 中文輸入法特有痛點，平台編輯器常見問題 |

#### 📚 書籍與章節管理

| 功能 | 說明 | 判斷依據 |
|------|------|---------|
| **多書管理（書架）** | 一個帳號管多本書，輕鬆切換 | 連載作者同時寫多本是常態 |
| **章節管理（拖拉排序）** | 章節自由排列，拖拉重組 | Scrivener Binder 最被稱讚的功能之一 |
| **章節狀態標記** | 初稿 / 修改中 / 已完成 / 存稿 | 連載作者管理存稿必需 |
| **全文搜尋** | 全書關鍵字搜尋 | 百萬字小說沒搜尋是噩夢 |

#### 📊 字數追蹤

| 功能 | 說明 | 判斷依據 |
|------|------|---------|
| **多維度字數統計** | 今日新增 / 本章 / 全書 / 選取段落 | 網文作者字數就是 KPI |
| **每日字數目標** | 設定目標 + 進度條顯示 | Scrivener 最被推崇的動力功能 |
| **字數歷史記錄** | 每日寫了多少的歷史圖表 | 讓努力「看得見」 |

#### 👤 世界觀輔助工具

| 功能 | 說明 | 判斷依據 |
|------|------|---------|
| **人物管理（角色卡）** | 人物基本資料、特徵、關係 | 百萬字小說記不住人物設定是真實痛點 |
| **研究資料附件** | 可貼入圖片、連結、筆記作為參考 | 歷史文 / 架空文需要大量考據資料 |

#### 📤 輸出

| 功能 | 說明 | 判斷依據 |
|------|------|---------|
| **基本匯出（TXT / DOCX / MD）** | 不被工具綁死；投稿、備份需求 | 平台投稿通常要 TXT 或 Word |

---

### ⭐ GOOD TO HAVE（差異化加分項，能顯著提升留存和口碑）

#### 🗂 結構規劃

| 功能 | 說明 | 對誰重要 |
|------|------|---------|
| **大綱工具（層級式）** | 卷 → 章 → 場景的階層規劃 | Plotter 派必需；Planster 會用 |
| **情節板（Corkboard）** | 索引卡式的場景拖拉重組 | Plotter + 修稿中用戶 |
| **章節摘要欄（Synopsis）** | 每章可寫一句話摘要 | 讓整體結構一眼可看 |
| **時間軸視圖** | 多線敘事或跨時間的場景排列 | 複雜結構小說作者 |
| **場景詳情（POV/地點/時間標注）** | 場景屬性標記 | 多視角小說作者 |

#### ✍️ 寫作輔助

| 功能 | 說明 | 對誰重要 |
|------|------|---------|
| **分割視窗** | 同時顯示大綱/人物資料 + 正文 | 進階用戶；減少視窗切換 |
| **版本快照（Snapshot）** | 儲存某章節的特定版本 | 大改前的保險 |
| **行內批注 / 評論** | 對自己或對 beta reader 留注 | 修稿流程 |
| **讀稿模式（預覽）** | 模擬讀者閱讀的排版預覽 | 最後校稿階段 |
| **寫作計時器（番茄鐘）** | 專注寫作 25 分鐘倒數 | 有拖延問題的作者 |
| **語音輸入整合** | 支援語音轉文字輸入 | 手機高頻用戶、通勤族 |

#### 🎯 動力與習慣養成

| 功能 | 說明 | 對誰重要 |
|------|------|---------|
| **連續寫作天數（Streak）** | 顯示連續幾天有達到目標 | 習慣養成需求者 |
| **里程碑設定** | 完成 X 萬字的慶祝提示 | 長篇作者的激勵機制 |
| **寫作提醒通知** | 每天設定固定寫作時間提醒 | 有拖延問題的作者 |

#### 🌍 世界觀深度工具

| 功能 | 說明 | 對誰重要 |
|------|------|---------|
| **人物關係圖** | 視覺化角色關係網絡 | 登場人物多的長篇 |
| **地圖/場景管理** | 可上傳地圖、標記地點 | 奇幻/架空歷史 |
| **設定 Wiki（雙向連結）** | 設定頁面之間可相互連結 | 複雜世界觀作者 |
| **標籤系統** | 自定義標籤分類任何內容 | 重度使用者 |

#### 📤 進階輸出

| 功能 | 說明 | 對誰重要 |
|------|------|---------|
| **EPUB 輸出** | 輸出標準電子書格式 | 自出版作者 |
| **自定義排版模板** | 出版社格式、稿紙格式等 | 傳統投稿作者 |
| **分享閱讀連結** | 分享給 beta reader 但不能下載 | 需要外部反饋的作者 |

---

### 🚫 暫不建議（資源有限時跳過）

| 功能 | 原因 |
|------|------|
| **平台直投整合** | 各平台 API 協議複雜，且平台規則常變 |
| **AI 寫作生成** | 已有大量競品做，難以差異化；且中文 AI 品質參差不齊 |
| **文法/錯字自動修正** | 中文語境複雜，誤判率高，容易引起反感 |
| **出版社投稿管理** | 受眾過小（傳統出版管道作者） |
| **社群/讀者互動功能** | 平台才有流量優勢；自建社群難贏平台 |

## 五、競爭差距分析：你的機會在哪裡

### 5.1 競爭對手的共同缺口

| 缺口 | 哪些競品沒做好 | 你的機會 |
|------|--------------|---------|
| **中文 UX 最佳化** | Scrivener 翻譯差、中文輸入問題多 | 原生中文體驗，標點符號對、縮排對 |
| **輕量 + 快速上手** | Scrivener 學習曲線嚇退新人 | 5 分鐘能開始寫作的起點 |
| **手機體驗** | Scrivener 手機版幾乎廢掉 | 手機與桌面端無縫的核心體驗 |
| **連載工作流** | 所有工具都預設「完成後出版」 | 存稿管理、更新節奏追蹤 |
| **雲端不鎖定** | 平台編輯器鎖在各自平台 | 開放格式 + 多平台匯出 |
| **美觀的沉浸感** | Scrivener 老、平台編輯器素 | 有設計感的寫作空間 |

### 5.2 核心定位建議

> 「**給認真寫作的華文作者，一個真正屬於他們的工作空間**——比 Scrivener 好上手，比 Notion 更懂小說，比平台編輯器更自由。」

## 六、數據摘要

- **全球小說寫作軟體市場**：Scrivener 仍是最高知名度，但 UI/行動端被大量抱怨
- **中文市場特殊性**：網文市場規模龐大（中國線上文學年收入超過 300 億人民幣），但缺乏一款「完整寫作工作空間」工具
- **作者平均每日碼字目標**：網文作者 3,000–5,000 字；傳統文學作者 500–1,500 字
- **跨裝置需求**：超過 70% 的手機端寫作行為發生在通勤 / 睡前 / 等待間隙
- **用戶最怕的事**：丟稿（第一位）、被打斷心流（第二位）、設定遺忘/矛盾（第三位）
